Data Sources and How They Are Used

Scope: This page documents where CieloRumbo gets its data, which parts are local vs external, how each source is used in the workflow, and what is cached for reuse across pages.

External Sources

  • AviationWeather.gov: METAR, TAF, station info, airport lookup, and fallback weather resolution.
  • FAA Class Airspace service: polygon geometry and numeric B/C/D/E shelf values for map and altitude-profile work.
  • FAA chart metadata: sectional and TAC reference lookup for chart-review workflows.
  • USGS elevation service: route elevation samples used to derive terrain-aware Class E/G behavior.
  • OpenStreetMap / OpenTopoMap: base maps for route review, terrain context, and checkpoint visualization.

Local / Offline Components

  • magvar: computes magnetic variation locally from the World Magnetic Model.
  • Aircraft profile JSON: contributor-managed aircraft configurations stored in the repo.
  • Checkpoint plans: browser-side saved plan reused between the home page, planner, and route map.
  • Nav-log snapshots: saved browser-side so the generated log can be reopened without recalculation.
  • FAA airspace cache: browser-side cache reused between home, route map, and airspace profile.

Weather

Weather and Forecast Data

Observed Conditions

When the selected flight time is current or in the past, the app uses FAA observation data from AviationWeather. The server normalizes METAR values into the fields the frontend needs for airport weather, nav-log calculations, and field-level status messages.

Future-Dated Planning

When the selected flight date/time is in the future, the app switches to FAA TAF forecast data where available. Wind and temperature come from the forecast segment, while the latest usable observation can still supply context like altimeter or location fallback when appropriate.

Navigation

Variation, Aircraft, and Checkpoints

Magnetic Variation

Magnetic variation is computed locally with magvar, which uses the same World Magnetic Model family NOAA relies on. No external key or runtime lookup is needed for that step.

Aircraft Profiles

Aircraft data is stored as JSON profile files inside the repository. The local backend exposes those files through a small CRUD API so the browser can select, edit, and save aircraft configurations.

Checkpoint Data

Checkpoints combine route geometry, curated visual-checkpoint data, airport-reference data, and map-derived candidate logic. Once approved, the saved checkpoint plan becomes the source of truth for the nav log, route map, and planner.

Airspace

Map and Altitude Profile Inputs

FAA Airspace Geometry

The route map and airspace profile both use FAA class-airspace geometry. For Class B, C, and most D shelves, the same feed also provides usable numeric vertical values such as LOWER_VAL, LOWER_CODE, UPPER_VAL, and UPPER_CODE.

Terrain and Class G

The altitude profile samples route elevation from USGS so surface-based and AGL-based floors can be interpreted meaningfully. That allows the chart to show terrain, convert some Class E floors to MSL, and derive Class G below the first controlled floor.

Caching

What Gets Reused

Browser Cache Reuse

Airspace, checkpoint plans, flight drafts, and nav-log snapshots are reused across page transitions. The home page now also prefetches FAA airspace after nav-log generation so later map/profile views start with less waiting.

Live-Only Data

Weather is intentionally refreshed live rather than saved into the reusable plan file. That keeps plan structure portable while avoiding stale operational weather data.

Limitations

Current Boundaries

  • FAA airspace map overlays

Still Evolving
