OpenFlight AI

VFR Flight Planning, Weather Ingestion, and Nav-Log Generation

OpenFlight AI is a browser-first cross-country planning tool for educational use. It combines weather retrieval, aircraft performance profiles, checkpoint planning, and route review into a single local workflow.

View Repository Explore Architecture

At a Glance

  • Frontend: Static HTML/CSS/JS
  • Backend: Node.js + Express
  • Weather: aviationweather.gov
  • Variation: offline WMM via magvar
  • Testing: Jest + Playwright

Disclaimer: This project is for educational purposes only. It should not be relied on as the sole source for real-world navigation planning, weather interpretation, or operational decision-making.

Overview

What the app does

Flight Setup

The home page collects departure, destination legs, cruise altitude, and aircraft selection. Weather loads per airport and populates the data needed for the navigation tables and printable nav log.

Checkpoint Planning

A separate planner generates and stores intermediate dead-reckoning checkpoints. Users can edit, add, remove, and save checkpoints before they are reflected back into Table 3 and the PDF workflow.

Route Map

The map page renders the active route on a free OpenStreetMap base layer, supports per-leg focus and visibility toggles, highlights saved checkpoints, and exports the route to KML for Google Earth review.

Aircraft Profiles

Aircraft performance data lives in separate profile files so contributors can add new aircraft without changing calculation code. The browser UI can list, edit, and save profiles through the local backend.

Frontend

public/js/app.js orchestrates the main nav-log workflow.

public/js/checkpoints.js manages checkpoint review and editing.

public/js/map.js handles route plotting, leg controls, and KML export.

Shared State

public/js/flightStore.js is the browser-side cache and contract between pages. It keeps flight drafts, checkpoint plans, and nav-log state synchronized across page transitions.

Backend

index.js exposes local endpoints for weather lookup, airport comms, checkpoint generation, and aircraft profile CRUD. Server-side normalization keeps third-party API quirks away from the UI.

Data Sources

What is local and what is external

External Services

  • AviationWeather: METAR, station data, airport lookup, and fallback weather resolution.
  • OpenStreetMap / OpenTopoMap: route and terrain map tiles on the route map page.

Local / Offline Components

  • magvar: computes magnetic variation locally from the World Magnetic Model.
  • Aircraft profile JSON files: contributor-managed data files under the aircraft profile directory.
  • Checkpoint plan cache: browser storage reused between home, planner, and map.

Reliability

How the app avoids brittle behavior

Weather Robustness

Weather lookups show per-airport loading states, retry transient upstream failures, and surface incomplete-data errors inline next to the field that triggered them.

Single Sources of Truth

Navigation logic, weather normalization, and aircraft profiles are separated so changes happen in one place instead of drifting across duplicated code paths.

Cross-Page Reuse

Once checkpoints or a nav log are generated, the stored plan is reused across pages instead of being recalculated every time the user navigates.

Testing

Local-first change safety

Unit Tests

Jest covers weather normalization, fallback behavior, magnetic variation, and supporting services. These tests verify pure logic and API adaptation behavior.

End-to-End Tests

Playwright validates the real user workflows: weather entry, checkpoint planning, nav-log generation, cross-page state reuse, and route map behavior. 

npm test
npm run test:unit
npm run test:ui
