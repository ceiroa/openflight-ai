CieloRumbo

VFR Flight Planning, Weather Ingestion, and Nav-Log Generation

CieloRumbo is a browser-first cross-country planning tool for educational use. It combines weather retrieval, aircraft performance profiles, checkpoint planning, route review, airspace overlays, and altitude profiling into a single local workflow.

Overview

What the app does

Flight Setup

The home page collects departure, destination legs, cruise altitude, and aircraft selection. Weather loads per airport and populates the data needed for the navigation tables and printable nav log.

Checkpoint Planning

A separate planner generates and stores intermediate dead-reckoning checkpoints. Users can edit, add, remove, and save checkpoints before they are reflected back into Table 3 and the PDF workflow.

Route Map

The map page renders the active route on a free OpenStreetMap base layer, supports per-leg focus and visibility toggles, highlights saved checkpoints, shows FAA airspace overlays, supports current-location display, and exports the route to KML for Google Earth review.

Airspace Profile

A dedicated airspace profile page projects the route against FAA airspace, samples terrain, derives Class G below the first controlled floor, and shows altitude structure from the surface up to the Class A floor.

Aircraft Profiles

Aircraft performance data lives in separate profile files so contributors can add new aircraft without changing calculation code. The browser UI can list, edit, and save profiles through the local backend.

Frontend

public/js/app.js orchestrates the main nav-log workflow.

public/js/checkpoints.js manages checkpoint review and editing.

public/js/map.js handles route plotting, leg controls, and KML export.

Shared State

public/js/flightStore.js is the browser-side cache and contract between pages. It keeps flight drafts, checkpoint plans, and nav-log state synchronized across page transitions, and also shares FAA airspace cache data between the home page, route map, and altitude-profile workflow.

Backend

index.js exposes local endpoints for weather lookup, airport comms, checkpoint generation, airspace lookup, elevation sampling, chart metadata, and aircraft profile CRUD. Server-side normalization keeps third-party API quirks away from the UI.

Data Sources

What is local and what is external

External Services

  • AviationWeather: METAR, TAF, station data, airport lookup, and fallback weather resolution.
  • FAA: class-airspace geometry and vertical shelf values, plus sectional chart metadata.
  • USGS: route elevation samples used for terrain and Class E/G altitude derivation.
  • OpenStreetMap / OpenTopoMap: route and terrain map tiles on the route map page.

Local / Offline Components

  • magvar: computes magnetic variation locally from the World Magnetic Model.
  • Aircraft profile JSON files: contributor-managed data files under the aircraft profile directory.
  • Checkpoint and nav-log cache: browser storage reused between home, planner, and map.
  • Airspace cache: shared browser-side FAA airspace cache reused by the route map and altitude profile.

Reliability

How the app avoids brittle behavior

Weather Robustness

Weather lookups show per-airport loading states, retry transient upstream failures, and surface incomplete-data errors inline next to the field that triggered them. Future-dated planning switches to FAA forecast handling when observation-only data is no longer appropriate.

Single Sources of Truth

Navigation logic, weather normalization, and aircraft profiles are separated so changes happen in one place instead of drifting across duplicated code paths.

Cross-Page Reuse

Once checkpoints or a nav log are generated, the stored plan is reused across pages instead of being recalculated every time the user navigates. The same pattern now applies to FAA airspace, which is prefetched and cached for later map/profile use.

Testing

Local-first change safety

Unit Tests

Jest covers weather normalization, fallback behavior, magnetic variation, and supporting services. These tests verify pure logic and API adaptation behavior.

End-to-End Tests

Playwright validates the real user workflows: weather entry, checkpoint planning, nav-log generation, cross-page state reuse, route map behavior, airspace caching, and the altitude-profile workflow. 

npm test
npm run test:unit
npm run test:ui

AI Workflow

How AI tools were used

Where AI helped

AI tools were used as implementation accelerators for UI changes, refactors, integration glue, test scaffolding, and exploratory feature work. That was especially useful while the app was expanding from a single nav-log screen into weather, checkpoints, mapping, and aircraft-profile workflows.

What made it reliable

The project became more stable only after the code was reorganized around clearer boundaries, shared browser state, standardized data files, and repeatable tests. In other words, the useful pattern was not just "ask an AI to change code," but "constrain the change, verify it locally, and keep one source of truth per area." 

Typical AI-assisted workflow:
1. make one bounded change
2. update or add tests in the same change
3. run npm test locally
4. push only after the workflow passes
